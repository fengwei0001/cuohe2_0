觅游方向二期：从撮合网络到 Agent Workforce

承接一期汇报 方向提案
日期：2026 年 7 月 ｜ 基于一期方向（撮合网络）落地验证后的进一步思考

一、一句话结论

一期我们达成共识：觅游从「纯 A2A 围观社区」转向「人 + Agent 的撮合型社交网络」，并以招聘（钳程无忧）作为第一块实验田。在实际推进中，我发现了一个更底层的判断：

撮合网络的核心瓶颈不是网络架构，而是单边价值不够强。双边网络不应是起点，而应是结果——先用 Agent 工具能力服务好一个边，这个边聚集起来后，另一边自然被吸引过来，网络自己长出来。

基于此，我想提出觅游的下一步方向：从「建一个 Agent 网络」转向「让 Agent 作为 workforce 部署到现实世界」——Agent 不是网络里的一个节点，而是出去替人干活的一个工人。先做工具，再长网络。

二、一期复盘：钳程无忧验证了什么、遇到了什么

2.1 验证了的

  • 「人 + Agent」的模式是成立的：用户一句话吩咐，虾完成建档→发布→盯机会→破冰的全链路，体验远好于「自己每天刷 Boss直聘」。
  • 隐私边界设计被认可：「服务端零 LLM、简历永不上行」的架构让用户安心。
  • Agent 能做好撮合前段：建档、脱敏、匹配、破冰——这些原本需要猎头做的事，Agent 能完成到足够好的程度。
  • 工程可行性全链路跑通：Agent 身份体系、心跳、粗筛、icebreak，从后端到 Agent 协议完整落地。

2.2 遇到的瓶颈

  • 双边冷启动：招聘是典型双边市场——求职者少则 HR 不来，HR 少则求职者没有匹配结果。即使 Agent 精排做到满分，池子里没有合适的 JD/简历，也无法兑现价值。
  • 与成熟平台的规模差距：Boss直聘有几千万简历和几百万 JD，觅游的招聘板目前无法在规模上竞争。
  • 用户的核心动作仍然在外部：真正的面试、谈薪、入职发生在觅游之外。Agent 的价值只覆盖了漏斗前段。

2.3 关键反思

钳程无忧的模式是「建一个封闭网络，让双边在里面匹配」。但更有效的路径也许是：先让 Agent 替单边用户出去外面的世界搜集信息、采取行动——不依赖网络内有人发布信息，而是主动到 Boss直聘、企业官网、拉勾、实习僧去获取。单边用户积累起来后，双边网络自然形成。

三、新判断：Agent 的价值不在网络里，在现实世界里

3.1 重新审视 EigenFlux 和撮合网络

EigenFlux（eigenflux.ai）是我们的参考项目之一——一个 Agent 通信/广播网络。它的核心假设是「Agent 之间需要一个共享信息层来交换信号」。但仔细分析后，这个假设面临一个根本性质疑：

Agent 能上网搜索、能读 RSS、能爬数据源、能调 API。它们不是信息孤岛。「Agent 独立搜索造成重复劳动」这个前提，在 LLM 调用成本持续暴跌的今天，痛感并不强。

EigenFlux 和我们一期的撮合网络，共同面临的问题是：它们都试图先建一个封闭网络，再让网络产生价值。但网络价值跟规模成正比，规模从零开始是最难的事。

3.2 真正高价值的方向：取代「很贵很不好用的中间人」

换一个角度思考：在现实世界中，有哪些「中间人」角色收费很贵、但服务质量参差不齐，而 Agent 可以做到「更好且更便宜」？

  • 中间人的价值确实存在（信息撮合 + 沟通代理 + 流程协调）；
  • 但人类中间人贵、不稳定、且大多数并不优秀；
  • Agent 能在信息量、响应速度、一致性、耐心、成本上碾压平均水平的人类中间人；
  • 只需做到「普通中间人」的水平，就能因为价格差 10-100 倍而切下大量市场。

3.3 核心洞察：Agent 不是「网络里的节点」，是「部署到世界里的工人」

Agent 的价值不来自「它在某个网络里能找到谁」，而来自「它能替人完成原本需要花钱雇人干的工作」。这跟编程 Agent（Cursor/Claude Code）的逻辑完全同构——Cursor 没有建「程序员社交网络」，它只是让写代码更快更便宜。

但有一点很重要——虽然我们从工具价值启动，但本质上依然是在做人与人之间的链接。

Agent 是人与人之间的中间节点，不是孤立的效率工具。一个求职 Agent 去爬了企业官网的招聘信息，看似是「工具在抓数据」，但它实际做的事是：替求职者与招聘者之间建立起了一条沟通通道。它在理解求职者的诉求、理解 JD 的要求、判断双方是否匹配、在合适的时机促成连接——这就是中间人/猎头的工作。

所以「从工具出发」和「做人与人的连接」并不矛盾：

工具能力是手段，人与人的连接是目的。Agent 先用工具能力（爬虫、匹配、话术）积累单边价值，但它最终促成的依然是人与人之间的撮合——求职者找到工作、买家买到车、租客找到房。只不过路径变了：不是先建一个空网络等双边来，而是先用工具服务好一边，让连接自然发生。

四、路径设计：工具 → 单边聚集 → 双边网络

Phase 1 工具服务单边 → Phase 2 单边规模化 → Phase 3 另一边被吸引 → Phase 4 双边网络

关键原则：每一步都独立有价值，不依赖下一步成立。即使永远停留在 Phase 1（工具层），也有独立的商业模式和用户价值。双边网络是结果，不是前提。

五、场景深挖：两条验证线

5.1 线一：求职 Agent（与钳程无忧衔接，mars 内增值）

Phase 1 — 单边工具

核心能力：Agent 替用户从外部世界获取招聘信息 + 求职辅导

  • JD 爬虫：从企业官网、Boss直聘公开页、拉勾、实习僧等平台持续抓取 JD，按用户画像匹配推送
  • 简历润色：针对具体 JD 定制简历版本（突出匹配点、调整表述）
  • 模拟面试：基于 JD 要求出题、模拟面试官追问、给出改进建议
  • 投递助手：替用户在各平台投递（可选，需授权）
  • 机会监控：持续心跳扫描新增 JD，有匹配时主动通知（静默规则）

用户侧价值：「我只需要告诉虾我想找什么工作，它帮我盯着全网、帮我改简历、陪我练面试。」

与钳程无忧的关系：钳程无忧的双边匹配保留为「内部加速通道」——当平台上同时有 HR 发布需求时，走内部直连（更快更精准）；但即使没有 HR 入驻，用户依然能通过爬虫获得外部 JD 的价值。

Phase 2-4 — 从单边到网络

求职者积累成池子 → 对 HR/企业有吸引力 → HR 入驻发布 JD → 双边直连效率高于爬虫 → 网络自然形成。此时钳程无忧的撮合引擎成为核心价值（内部直连 > 外部爬取）。

5.2 线二：销售 Agent（独立商业化，B 端商家）

Phase 1 — 单边工具

核心能力：Agent 替商家做销售接待、报价、跟进、促单

  • 渠道接入：部署到商家已有阵地（企业微信账号），客户加好友后直接跟 Agent 对话——客户不需要知道、也不需要下载新 App
  • 知识库驱动：商家录入商品/服务信息（如车源库、菜单、服务项目），Agent 自动生成话术、回答咨询
  • 行为规则引擎：商家设定底价、服务范围、营业时间、何时转人工
  • 主动跟进：对意向客户定期问候、新品到货推送、促销活动通知
  • 语音能力：支持语音消息甚至电话接入（ASR → LLM → TTS）

商家侧价值：「相当于雇了一个 7×24 在线、无限耐心、知识全面、同时服务 100 个客户的金牌销售，月费只要几百块。」

单位经济对比：

人类销售销售 Agent
月成本1.5-2.5 万（底薪+提成+管理）几百到几千（按对话量/月费）
同时服务客户3-5 个无限
在线时间8-10 小时24 小时
知识覆盖经验依赖，参差不齐全品类知识库，一致且可更新
情绪/态度波动、疲劳、离职永远耐心、永远在线
Phase 2-4 — 从工具到网络

以二手车为例：车商使用销售 Agent 接客 → 车商越来越多 → C 端买车用户发现「找车直接问这些 Agent 比逛 58 体验好」→ 用户主动来平台找 Agent 询价 → 交易网络自然形成（类似闲鱼，但服务层更厚）。

关键锁定效应：商家一旦用起来（灌了知识库、跑通了话术、积累了客户对话数据），迁移成本极高。这是闲鱼没有的壁垒——闲鱼的工具层很薄（拍照+发布），供给侧没有切换成本。

六、为什么必须做「云虾」，以及为什么我们能做

6.1 本地 Agent 是一条走不通的路

钳程无忧一期的设计里，Agent 运行在用户本地（OpenClaw / Claude Code 等），靠心跳轮询来执行任务。这个模式对技术用户（开发者、AI 从业者）是可行的——他们本来就在用这些工具。

但觅游要服务的用户不是这群人：

  • 求职者：大量是在校学生、应届生——让他们安装 CLI、配置 Agent 运行环境、保持本地进程在线，门槛太高。
  • 中小商家：二手车商、餐饮老板、房产中介——他们连 CRM 都不一定会用，更不可能维护一个本地 Agent 运行时。

如果 Agent 要成为大众可用的劳动力，它必须跑在云端。用户不需要懂技术、不需要装软件、不需要保持电脑在线。他们只需要通过自己熟悉的渠道（微信、电话、网页）跟 Agent 交互，Agent 在云端 7×24 运行、自动执行任务。

6.2 自研 Longcat 让「云虾」在能力和成本上都可行

做云端 Agent 的两个核心约束是推理能力和推理成本。好在美团有自研大模型 Longcat：

  • 能力层面：Longcat 具备多轮对话、意图识别、工具调用、长上下文理解等能力，足以支撑销售接待、求职辅导、信息比较等场景。
  • 成本层面：自研模型的推理成本远低于调用外部 API（GPT-4 / Claude），这让「每个商家月付几百块雇一个 AI 销售」的单位经济模型成立——如果用外部 API 则成本可能倒挂。
  • 可控性：自研意味着可以针对具体场景做 fine-tune（销售话术、行业知识），不受第三方模型策略变更影响。

「云虾」= 觅游云端托管的 Agent 实例，底层跑 Longcat 推理，上层接入品类知识库和行为规则，通过企微/电话/网页触达终端用户。对用户和商家而言，它就是一个「随时在线、什么都懂的员工」，不需要理解背后的任何技术。

七、觅游的终局定位

觅游的终局依然是「人 + Agent 的社区与撮合网络」——这个方向没有变。变的是到达那里的路径：

不是先建一个空的社区/网络等人来，而是先用 Agent 工具能力让用户觉得「虾真有用」，用户和商家因为工具价值聚集起来之后，社区有了真实内容、撮合网络有了双边供给——它们就不再是需要硬拉的冷启动产品，而是自然生长出来的生态。

三者之间的关系：

  • Agent 工具能力：启动引擎——让用户留下来的第一个理由（「虾能帮我找工作 / 帮我接客」）
  • Agent 社区：沉淀层——用户和 Agent 的经验、内容、口碑在社区里沉淀，形成发现和分享的场
  • 撮合网络：价值放大器——当双边都在平台上时，直连匹配比外部爬虫更快更精准，网络效应成为壁垒

这与美团的核心基因高度契合：

  • 美团的使命：帮大家吃得更好、生活更好 — 连接人与服务，促成交易
  • 觅游的角色：帮商家更好地服务用户 — 用 Agent 替代人力中间层，降低成本、提升效率
  • 切中的结构性机会：本地生活商家的数字化程度低、人效低。一个小商家雇不起好销售，但月付几百块「雇」一个 Agent，这就是实在的降本增效。

九、下一步建议

  1. 选定第一个验证场景：建议从两条线中选一条先做 MVP：
    • 求职 Agent（与钳程无忧最近、面向觅游现有用户、改动小、验证快）
    • 销售 Agent（商业模式最清晰、ROI 可量化、但需要企微接入+语音能力）
  2. 定义 PMF 指标：
    • 求职 Agent：「用户留存率 + 推荐 JD 的点击/投递率 + 面试通过率提升」
    • 销售 Agent：「客户响应率 + 成交转化率 + 商家续费率 + vs 人工销售的成本对比」
  3. 明确与 Longcat 的协作模式：觅游调用 Longcat 做底层推理，还是自建垂直模型？建议前期直接调 Longcat，降低启动成本。
  4. 设定 3 个月里程碑：选定场景后，3 个月内跑出 Phase 1 的工具层 MVP，验证单边价值是否足够强。

十、论证小结

  1. 一期验证：「人 + Agent 撮合」方向成立，但双边冷启动是核心瓶颈。
  2. 路径修正：不应先建网络再等价值出现，而应先做单边工具价值，让网络自然生长。
  3. 价值定位：Agent 取代「很贵很不好用的人类中间人」，这是确定有价值且可量化的市场。
  4. 路径公式：工具 → 单边聚集 → 双边网络 → 交易平台。每步独立有价值。
  5. 与 Longcat 互补：Longcat 提供引擎，觅游做场景化落地，定位不冲突。
  6. 一期资产全复用：钳程无忧建设的身份/心跳/匹配/协议能力是 Agent Workforce 的基础设施。
  7. 终局：Agent Workforce 平台 — 帮商家「雇」Agent 干活，平台自然长成交易网络。契合美团本地生活基因。

以上是基于一期方向落地验证后的进一步思考，核心逻辑：先做工具 → 再长网络。欢迎团队讨论指正。
